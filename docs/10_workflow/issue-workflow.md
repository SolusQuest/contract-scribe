# Issue workflow

Use native GitHub issue types. Split work when it needs an independent acceptance contract, dependency state, pull request, or review cycle.

Every implementation issue must state its goal, bounded scope, acceptance criteria, validation, dependencies, and relevant pinned documentation links. Use repository-local parent/sub-issue relationships for decomposed work; use full URLs for cross-repository traceability.

M0 experiments must report observed evidence and unresolved risks. They must not turn an untested architecture option into a project-wide assumption.
