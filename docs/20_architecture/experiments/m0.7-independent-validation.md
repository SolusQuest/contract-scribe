# M0.7 independent validation protocol

This record defines the M0.7 protocol for Issue #10. It validates the unchanged M0.6-selected framework-dependent semantic execution baseline against the public fixture [Yuee98/contract-scribe-m07-fixture](https://github.com/Yuee98/contract-scribe-m07-fixture) at commit [`aee85e30a7634fdf6adce7ac8b1a185a68b9698a`](https://github.com/Yuee98/contract-scribe-m07-fixture/commit/aee85e30a7634fdf6adce7ac8b1a185a68b9698a). The fixture manifest at that commit records the owner, public availability expectation, pre-run independence/public-safety review, protected file inventory, and SHA-256 values. The fixture source and canonical `expected-payload.json` oracle were committed before selected-baseline execution, with no trailing newline, and `.gitattributes` pins LF text checkout across required cells.

The selected baseline is ContractScribe commit [`645c0946b8b811d633b471b232b0654c10e6d7f6`](https://github.com/SolusQuest/contract-scribe/tree/645c0946b8b811d633b471b232b0654c10e6d7f6), with semantic source revision `63fd9a0ab5ff33ae20d8f7b9e66714a96feea39e` and transfer-manifest SHA-256 `1ad92419cf77e82f660cd1226f2a10dbb0561b6271b0bfc482a8bb4799237c7f`. M0.7 obtains a clean checkout of that commit, runs its unchanged M0.4 protocol/provenance checks there, and invokes its already-built host against the independent fixture. New M0.7 files are never admitted into the old M0.4 closed provenance file set.

The required cells are `ubuntu-latest` and `windows-latest` with X64 process architecture, SDK `10.0.102` under `latestFeature` roll-forward, the current M0.4 package baseline including `System.Security.Cryptography.Xml` `9.0.18`, and observed framework-dependent RIDs `linux-x64` and `win-x64`. The fixture preparation commands are `dotnet restore Sample.sln` followed by `dotnet build Sample.sln --configuration Release --no-restore`. Selected-baseline execution begins only after preparation and uses two fresh host processes per cell without network, credentials, or write tokens.

The independent oracle is the complete canonical payload. Each output must be UTF-8 without BOM, contain no trailing newline, match the oracle byte-for-byte, match the other fresh process in its cell, and match the other required cell. The verifier writes a bounded evidence record containing only identities, hashes, toolchain fields, RIDs, commands, comparison results, outcome classifications, CI identity, and unresolved risks.

The lifecycle distinguishes the selected-baseline commit, semantic source revision, M0.7 protocol commit, evidence-generating CI run, evidence-publication commit, fixture/oracle commit, and later ADR-promotion commit. ADR 0001 remains provisional in the implementation PR; after merge and a successful post-merge main run, a separate follow-up PR promotes the ADR. Outcome classification is deterministic and non-overlapping: selected-baseline commit, semantic-source, transfer-manifest, frozen-host, runner, serializer, identity, or selected-baseline adaptation/drift is `baseline-invalidated` and returns the decision to M0.6; fixture, oracle, protocol-input, public-safety, or post-run evidence invalidation is `protocol-failure`; conforming semantic mismatch or nondeterminism is `baseline-failure`; missing or incompatible environment, cancellation, timeout, incomplete execution, or missing evidence before post-run validation is `inconclusive`; only complete positive evidence is `succeeded`. Aggregate precedence is `baseline-invalidated`, `protocol-failure`, `baseline-failure`, `inconclusive`, then `succeeded`, so every multi-cell result has one outcome.
